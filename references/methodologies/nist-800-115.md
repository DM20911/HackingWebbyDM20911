# NIST SP 800-115 — Technical Guide to Information Security Testing and Assessment

## Origen y autoridad
Publicada por el National Institute of Standards and Technology (NIST) en septiembre 2008. Es la guía **oficial del gobierno de EE.UU.** para evaluaciones técnicas de seguridad. Su valor es **legal y de cumplimiento**: si trabajas con banca, sector público o industrias reguladas, esta es la referencia citable.

Documento: https://nvlpubs.nist.gov/nistpubs/Legacy/SP/nistspecialpublication800-115.pdf

## Estructura: 4 fases

### 1. Planning
- Identificar **objetivos** del assessment (compliance, due diligence, post-incident).
- Definir **alcance técnico** (sistemas, redes, aplicaciones).
- Acordar **reglas de engagement** (ROE) por escrito.
- Determinar **tipo de assessment**: técnico vs no-técnico, anunciado vs no-anunciado, externo vs interno.
- Coordinación con stakeholders: legal, SOC, IT operations, business owners.

### 2. Discovery
Dividida en dos:

**2a. Passive (sin tocar el target):**
- OSINT (whois, DNS, Google dorks, crt.sh, archive.org).
- Document review (políticas, arquitectura).
- Network sniffing (si autorizado en red interna).

**2b. Active:**
- Port scanning (nmap).
- Service enumeration.
- Vulnerability scanning automatizado (Nessus, OpenVAS, nuclei).
- Banner grabbing.
- Wireless discovery (si aplica).

### 3. Attack (Gaining Access)
Para cada vulnerabilidad confirmada:
- **Gaining Access:** explotar vulnerabilidad con scope autorizado.
- **Escalating Privileges:** verticales (user→admin) u horizontales (otro usuario).
- **System Browsing:** mapear lo accesible desde el punto comprometido.
- **Install Additional Tools:** solo si está autorizado (red team).

NIST hace explícito que el atacante debe documentar **cada step** y mantener artefactos de evidencia con timestamp.

### 4. Reporting
NIST exige estructura formal:
- Executive Summary (no técnico).
- Methodology used.
- Findings detallados con criticidad.
- Remediation recommendations.
- Evidence appendix.
- Lessons learned (proceso de testing en sí).

## Cómo aplicar NIST 800-115 a una auditoría web

| Fase NIST | Acción concreta |
|-----------|------------------|
| 1. Planning | Documento de scope firmado, ROE con ventanas, contactos SOC, criterios de éxito, plan de comunicación de incidentes. |
| 2a. Discovery passive | OSINT con subfinder/amass + búsqueda en repos públicos + dorks Google. |
| 2b. Discovery active | nuclei + httpx + ffuf + nikto. Inventario de tech stack. |
| 3. Attack | Por cada vuln: PoC con timestamp + log de cada request/response. Conservar como evidencia con hash SHA-256. |
| 4. Reporting | Informe NIST-style: ejecutivo + técnico + apéndices. Usar `scripts/generate_report.py` con plantilla DOCX corporativa. |

## Particularidades de NIST 800-115
- **Trazabilidad de evidencia**: cada artefacto con timestamp, hash, herramienta usada y operador.
- **Reproducibilidad obligatoria**: otra persona debe poder repetir cada finding con tu reporte solo.
- **Coordinación con SOC**: NIST asume que el equipo defensivo está al tanto y participa.
- **Métricas requeridas**: cobertura del alcance, density of findings, MTTR estimado.

## Tipos de assessment según NIST 800-115

| Tipo | Descripción | Cuándo |
|------|-------------|--------|
| Review Technique | Análisis de documentación, arquitectura, logs | Compliance audit |
| Target Identification | Discovery passive + active | Mapping de superficie |
| Target Vulnerability Validation | Scanning + manual verification | Vulnerability assessment |
| Penetration Testing | Discovery + Attack + Reporting | Full pentest |
| Information Security Test & Evaluation | Combinación con auditoría interna | Programa de seguridad |

## Cuándo usar NIST 800-115
- **Sector regulado** (banca, salud, gobierno, energía).
- **Auditorías citables** para cumplimiento (SOC 2, PCI DSS, HIPAA, ISO 27001).
- **Contratos con cláusulas de NIST** explícitas.
- Cuando el cliente exige formato compatible con su programa de gestión de riesgo.

## Cuándo NO es suficiente
- NIST 800-115 es **general** (no específica de web/API). Combinar con OWASP WSTG.
- No cubre detalles técnicos de explotación moderna (XSS DOM, SSRF cloud). Combinar con WSTG + ATT&CK.
- No es ideal para bug bounty (formal demás).

## Combinaciones recomendadas
- **NIST + OWASP WSTG** → técnico web + framework regulado.
- **NIST + PTES** → ambos compatibles, PTES da la granularidad y NIST da la citabilidad legal.
- **NIST + ATT&CK** → red team con cumplimiento NIST.
